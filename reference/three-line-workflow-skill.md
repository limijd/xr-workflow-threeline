# 三线流战法 Braided Workflow

用 AI 辅助开发不熟悉领域的项目时，三条主线绞合推进、边战边学的系统化方法。

## 核心理念

**不是先学再做，不是做完再总结，而是三线绞合、每一步同时产出。**

传统路径：学习 → 规划 → 开发 → 交付（串行，慢）
三线流战法：学习 ⊗ 开发 ⊗ AI驾驭 → 同步推进（绞合，快）

三条主线：

| 主线 | 代号 | 核心问题 | 产出物 |
|------|------|----------|--------|
| **Harness Engineering** | `HE` | 怎样驾驭 AI 来做这件事？ | 提示词模板、项目规范文档、工作流配置、skill |
| **领域知识** | `DK` (Domain Knowledge) | 这个领域的关键知识是什么？ | 知识地图、学习笔记、标准/规范索引 |
| **项目交付** | `DEL` (Delivery) | 项目本身怎么推进？ | 代码、文档、架构设计、可交付成果 |

**关键区分：三线是绞合（braided）而非并行（parallel）。**
并行 = 三条线各跑各的，偶尔同步。
绞合 = 每个工作单元都同时推进三条线，互相增强。

例如：在让 AI 帮你写一个 SLM 架构方案（DEL）的过程中，你同时在学习半导体退化物理（DK），并且在打磨让 AI 产出高质量技术方案的提示词策略（HE）。

## 命令

### `启动三线流 <项目描述>`

启动一个新的三线流项目。AI 执行战前侦察，生成三线地图。

### `三线状态`

查看当前三线推进状态和建议的下一步。

### `战斗日志`

生成当前 sprint/session 的三线进展记录。

### `切线 HE|DK|DEL`

手动切换当前焦点主线（通常 AI 会自动感知，但用户可以强制切换）。

### `复盘`

项目完成或阶段结束时，提炼三线的可复用成果。

## 阶段一：战前侦察

用户提供项目描述后，AI 生成**三线地图**：

### 1. 项目评估

```
项目名称: <name>
领域熟悉度: ★☆☆☆☆ ~ ★★★★★（用户自评或 AI 初判）
预估复杂度: <低/中/高>
AI 依赖度: <哪些环节重度依赖 AI，哪些用户可以独立完成>
```

### 2. HE 线规划（Harness Engineering）

```
AI 工具链:
  - 主力工具: <Claude Code CLI / Codex CLI / 其他>
  - 辅助工具: <web search / skill / MCP 等>

提示词策略:
  - 初期: <探索型提示，让 AI 帮助理解领域>
  - 中期: <执行型提示，让 AI 产出交付物>
  - 后期: <验证型提示，让 AI 帮助 review 和优化>

项目规范文档（待创建）:
  - CONVENTIONS.md: 项目约定（命名、结构、风格）
  - Claude.md / codex.md: AI 协作规范
```

### 3. DK 线规划（Domain Knowledge）

```
知识地图:
  ├── 必须掌握（Blocker）: <不懂就无法推进的核心概念>
  ├── 应该了解（Important）: <显著影响交付质量的知识>
  └── 可以后补（Nice-to-have）: <锦上添花的深入理解>

学习策略:
  - 即时学习: 遇到时让 AI 讲解，边做边学
  - 专题研究: 对 Blocker 级知识安排专门的学习 session
  - 标准/规范: 需要查阅的标准文档列表

预期产出:
  - 领域术语表
  - 关键概念笔记
  - 标准/规范索引（如适用）
```

### 4. DEL 线规划（Delivery）

```
里程碑:
  M1: <最小可行产出，证明路径可行>
  M2: <核心功能完成>
  M3: <完整交付>

技术栈选择:
  - <基于 AI 友好度和领域适配度选择>

交付物清单:
  - <具体的文件/代码/文档列表>
```

## 阶段二：战斗推进

### 工作单元结构

每个工作单元（session/sprint）遵循以下节奏：

```
1. 开场：三线状态检查（30秒）
   - 上次停在哪？
   - 三线各自的进度？
   - 本次 session 的重点？

2. 执行：绞合推进（主体时间）
   - 以 DEL 线任务为牵引
   - 遇到知识盲区 → 自动切入 DK 线
   - 发现 AI 协作模式可优化 → 记录到 HE 线
   - AI 感知当前瓶颈在哪条线，动态调整

3. 收尾：战斗日志（2-3分钟）
   - 本次 session 三线各推进了什么
   - 发现了什么（知识、技巧、问题）
   - 下次 session 的建议起点
```

### AI 自适应权重

AI 根据项目阶段和用户状态动态调整三线权重：

| 阶段 | HE 权重 | DK 权重 | DEL 权重 | 说明 |
|------|---------|---------|----------|------|
| 项目初期 | ██░░ | ████ | █░░░ | 重点建立知识基础和 AI 协作模式 |
| 进入状态 | █░░░ | ██░░ | ████ | 重点交付，知识按需补充 |
| 攻坚阶段 | ██░░ | ███░ | ███░ | 三线齐头并进，啃硬骨头 |
| 收尾阶段 | █░░░ | █░░░ | ████ | 全力冲刺交付 |

### 瓶颈检测与自动切线

AI 通过以下信号检测瓶颈所在的主线：

**DK 瓶颈信号**（需要切入学习）：
- 用户对 AI 产出无法判断对错
- 用户反复要求 AI 解释某个概念
- 交付物中出现领域常识性错误
- 用户说"这个我不太懂"、"这样对吗"

**HE 瓶颈信号**（需要优化 AI 协作）：
- AI 产出质量反复不达标，需要多轮修正
- 用户开始对 AI 的回答不满意
- 同类任务每次都要重新解释上下文
- 用户说"AI 老是理解错"、"效率太低了"

**DEL 瓶颈信号**（需要加速交付）：
- 知识和工具都就位但进度慢
- 过度学习、迟迟不动手
- 里程碑延期

## 阶段三：战后复盘

项目完成或阶段性节点时，执行复盘，提炼可复用资产：

### HE 线复盘
```
有效的提示词模式:
  - <哪些提示策略效果好>
  - <哪些踩了坑>

AI 协作规范沉淀:
  - <值得固化成 CLAUDE.md / codex.md 的规则>

Skill 候选:
  - <哪些重复性工作流可以抽象成 skill>
```

### DK 线复盘
```
知识资产:
  - <术语表、概念图、标准索引等>
  - <哪些知识是通用的，哪些是项目特定的>

认知升级:
  - <对这个领域的理解从哪里到了哪里>
  - <哪些认知被颠覆了>
```

### DEL 线复盘
```
交付物清单:
  - <最终产出了什么>

技术决策回顾:
  - <哪些选择是对的，哪些需要改>

可复用组件:
  - <代码、模板、架构模式>
```

## AI 行为准则

### 教学模式 vs 干活模式

AI 需要在两种模式间流畅切换：

**教学模式**（DK 线激活时）：
- 解释概念时联系项目实际场景，不做脱离语境的泛泛科普
- 给出"刚好够用"的知识深度，不过度展开
- 用用户已有的知识框架做类比（但注意：类比要基于解释效果，不要强行关联用户背景）
- 教完立刻接回 DEL 线的当前任务

**干活模式**（DEL 线激活时）：
- 专注产出，减少解释
- 遇到需要用户决策的领域问题，简要说明 tradeoff 而非长篇讲解
- 代码注释中嵌入关键知识点（让 DK 线搭 DEL 线的便车）

**HE 模式**（优化 AI 协作时）：
- 反思当前的提示词策略是否有效
- 建议创建或更新项目规范文档
- 识别重复性工作流，建议抽象为 skill 或模板

### 不做什么

- **不做串行切换**：不说"我们先学完这个再开始做"，而是在做的过程中学
- **不做过度学习**：DK 线的知识获取以 DEL 线需求为牵引，不为了"全面了解"而拖延交付
- **不做无记录推进**：每个 session 必须有战斗日志，三线进展必须可追溯
- **不做单线忽视**：即使某条线当前权重低，也要在战斗日志中记录其状态

## 适用场景

**最适合：**
- 用 AI 进入一个不熟悉的技术领域做项目
- 有明确交付目标但缺乏领域知识
- 希望系统性地积累 AI 辅助开发经验

**也适用于：**
- 熟悉领域但使用新工具链/框架
- 技术评估（如评估一个开源工具链是否值得投入）
- 写技术方案/PPT（领域知识 + AI 辅助 + 交付物三线并进）

**不太适合：**
- 完全熟悉的领域做常规开发（不需要 DK 线）
- 纯学习无交付目标（不需要 DEL 线）
- 一次性简单任务（杀鸡用牛刀）

## 与其他工具的集成

### Claude Code CLI / Codex CLI

三线流战法主要配合 Claude Code CLI 和 Codex CLI 使用：

- **CLAUDE.md / codex.md**: HE 线的核心产出物，将 AI 协作规范固化
- **Skill 系统**: HE 线的高级产出，将重复性工作流抽象为可复用的 skill
- **Plan Mode**: 战前侦察和复盘阶段使用 plan mode 进行全局规划

### 项目文件结构建议

```
<project>/
├── CLAUDE.md              # HE 线产出：AI 协作规范
├── THREELINE_MAP.md       # 三线地图
├── BATTLE_LOG.md          # 战斗日志
├── RETROSPECTIVE.md       # 复盘报告（阶段性/项目完成时）
├── docs/
│   └── knowledge/         # DK 线产出：知识笔记
├── src/                   # DEL 线产出：项目代码
└── ...
```
